MP1 ‚Äď Product / Service CRUD
============================================================

1) Product/Service CRUD (create/read/update/delete products)
   - File: admin/products.php (lines ~10‚Äď121, 133‚Äď400)
   - How it works:
     ‚ÄĘ Uses requireAdmin() to restrict access to admins only.
     ‚ÄĘ Handles Add product: POST `add_product` block inserts into `products` with prepared statement and calls `updateProductStatus`.
     ‚ÄĘ Handles Update product: POST `update_product` block updates product fields via prepared statement and keeps or replaces image.
     ‚ÄĘ Handles Delete product: POST `delete_product` block deletes from `products` using prepared statement.
     ‚ÄĘ Product listing and management UI are rendered in the same file (table of products with Edit/Delete actions).

2) Product/Service CRUD ‚Äď upload a single photo
   - File: admin/products.php (lines ~22‚Äď38 for add, 67‚Äď93 for update)
   - Related helper: config/functions.php ‚Üí `uploadPhoto` is NOT used here; image is handled manually.
   - How it works:
     ‚ÄĘ On add: checks `$_FILES['image']`, validates extension (jpg, jpeg, png, gif, webp), builds unique filename, moves file to `../products/`, then stores relative path in `products.image_url`.
     ‚ÄĘ On update: loads existing image, optionally uploads a new image with same validation, deletes old file with `unlink()` if replaced, then updates `image_url`.
     ‚ÄĘ Frontend display: index.php & products.php & product-detail.php read `products.image_url` and show the image if `file_exists()`.
     ‚ÄĘ This satisfies ‚Äúsingle main photo per product‚ÄĚ.

3) Product/Service CRUD ‚Äď upload/view multiple photos
   - Schema:
     ‚ÄĘ File: thepopstop.sql ‚Üí table `product_photos` (lines ~46‚Äď56).
       - `product_photos(product_id, photo_url, is_primary, display_order)` with FK to `products` ON DELETE CASCADE.
   - Admin upload & management:
     ‚ÄĘ File: admin/product-photos.php (lines ~21‚Äď71, 73‚Äď100, 100+ for UI)
     ‚ÄĘ How it works:
       - Validates `product_id`, loads product, then handles `upload_photos` POST.
       - Accepts multiple files via `$_FILES['photos']`, loops through each temp file.
       - Validates extension, stores file under `../products/` with unique name.
       - Inserts one row per photo into `product_photos` with incremental `display_order` and `is_primary` flag.
       - Handles delete: fetches `photo_url` by `photo_id`, deletes DB row and physical file.
   - Frontend viewing (gallery):
     ‚ÄĘ File: product-detail.php (lines ~1‚Äď120 for queries, ~256‚Äď299, 283‚Äď296, 259‚Äď281 for display)
     ‚ÄĘ How it works:

   - How it works:
     ‚ÄĘ Uses requireAdmin() so only admins can access.

     ‚ÄĘ Update user: POST `update_user` ‚Äď updates fields; optionally updates password if provided.

   - File: register.php (logic lines ~16‚Äď66, form lines ~90‚Äď160+)

3) User Update Profile
   - File: profile.php (lines ~12‚Äď91, UI from ~96+)
   - How it works:
     ‚ÄĘ Protected by requireLogin() ‚Äď only logged-in users can edit.

     ‚ÄĘ On POST, validates `full_name` and optionally password change.
     ‚ÄĘ If changing password: verifies current password with `password_verify`, checks length and confirm match, then updates hashed password along with profile fields.
     ‚ÄĘ If not changing password: updates full_name, phone, address, profile photo only.

4) User Registration ‚Äď upload a photo
   - File: register.php (lines ~46‚Äď55)
   - Helper: config/functions.php ‚Üí `uploadPhoto` (lines ~185‚Äď210)

     ‚ÄĘ `uploadPhoto` validates extension, creates directory if needed, builds unique name, moves uploaded file, and returns path.
     ‚ÄĘ Path stored in `users.profile_photo` at registration.

5) User Update Profile ‚Äď upload photo
   - File: profile.php (lines ~31‚Äď43)
   - How it works:
     ‚ÄĘ For profile update, reads existing `profile_photo` from DB.
     ‚ÄĘ If a new `profile_photo` is uploaded (no error), calls `uploadPhoto` with `uploads/profiles/` directory.
     ‚ÄĘ If upload succeeds, deletes old photo file (if it exists) and replaces `profile_photo` path.

6) Admin can deactivate a user

     ‚ÄĘ In the users table, shows an Activate/Deactivate button whose label and color depend on `is_active`.
     ‚ÄĘ Login logic (login.php lines ~35‚Äď38) checks `is_active` and blocks login for deactivated users with an error message.

7) Admin updates an active user‚Äôs role
   - File: admin/users.php (lines ~32‚Äď48 for update logic, ~369‚Äď377 for role select in edit modal)
   - How it works:
     ‚ÄĘ Admin can edit user via modal: role dropdown (`customer` or `admin`).

     ‚ÄĘ Prepared statement UPDATE sets `role` for the given `user_id`.

============================================================

1) Completed transaction uses prepared statements
   - File: checkout.php (lines ~41‚Äď116)
   - How it works:
     ‚ÄĘ On POST, sanitizes shipping address, payment method, discount code.
     ‚ÄĘ Validates discount using prepared statement on `discounts`.
     ‚ÄĘ Begins a DB transaction: `$conn->begin_transaction()`.
     ‚ÄĘ Inserts into `orders` using prepared statement.
     ‚ÄĘ For each cart item, inserts into `order_items` with prepared statement and updates product stock via another prepared statement, then calls `updateProductStatus`.
     ‚ÄĘ Deletes cart rows for the user via prepared statement.
     ‚ÄĘ Commits transaction; on any exception, rolls back and sets error.

2) Admin can view all orders and update status of the transaction
   - File: admin/orders.php (lines ~9‚Äď73, 134‚Äď207)
   - How it works:
     ‚ÄĘ Protected by requireAdmin().
     ‚ÄĘ Uses prepared statements to update `orders.status` when POST `update_status` is submitted.
     ‚ÄĘ Lists all orders (or filtered by status) with joined user data and item counts.
     ‚ÄĘ UI shows each order with customer, email, date, items, total, current status, and actions (View, Update).
     ‚ÄĘ ‚ÄúUpdate‚ÄĚ button opens a modal where admin changes status and submits the update form.

3) Send an email to the customer of the updated transaction (Mailtrap, list of products, subtotal & grand total)
   - Email composition & sending:
     ‚ÄĘ File: config/functions.php ‚Üí `sendOrderEmail` (lines ~212‚Äď389)
     ‚ÄĘ How it works:

         ¬∑ Product name, quantity, unit price, and line subtotal for each item.

         ¬∑ Order status and shipping address.
       - Uses PHPMailer with Mailtrap SMTP configuration (sandbox host, username, password, port 2525) to send email to the customer.
       - Fallbacks: can write HTML email content to `/logs/emails` for testing if PHPMailer isn‚Äôt configured.
   - Trigger point:
     ‚ÄĘ File: admin/orders.php (lines ~18‚Äď41)

   - File: thepopstop.sql (entire schema, especially lines ~7‚Äď204)
   - How it works:
     ‚ÄĘ Separate tables for entities: `users`, `products`, `orders`, `order_items`, `cart`, `discounts`, `reviews`, `suppliers`, `purchase_orders`, `purchase_order_items`, `email_notifications`, `expense_categories`, `expenses`.

     ‚ÄĘ Non-key attributes depend only on the key for each table (e.g., product attributes depend on product_id, order attributes on order_id, etc.), satisfying 2NF.

   - File: thepopstop.sql (lines ~206‚Äď231)
   - How it works:
